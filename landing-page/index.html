TV News Auto-Labeling Pipeline

Automated detection of commercial breaks, news segments, and teasers in TV news broadcasts using black frame analysis, AI transcript segmentation, and AWS serverless architecture

AWS Lambda Step Functions OpenCV Bedrock Claude React Amplify

Project Overview

This system automatically identifies commercial break segments within hour-long TV news broadcasts from MSNBC, CNN, and Fox News. It uses computer vision to detect the black frames that bookend commercial breaks, then pairs them into segments using timing heuristics based on standard commercial break durations.

🎬

Video Processing

Processes hour-long broadcasts by splitting them into 60-second chunks analyzed in parallel across dozens of Lambda workers.

🔍

Black Frame Detection

Uses color histogram analysis to identify black frames with high accuracy, distinguishing true commercial boundaries from dark scenes.

🧩

Intelligent Pairing

Pairs detected black frame events into commercial segments using standard duration heuristics (multiples of 30 seconds).

📊

Interactive Results

Web application for viewing, editing, and validating segmentation results with timeline visualization and video playback.

Backend Architecture

The processing pipeline runs entirely on AWS serverless infrastructure, deployed with SAM (Serverless Application Model).

1

Batch Orchestrator

Shell Script + AWS CLI

Lists all .mp4 videos in S3, groups them into batches of 20 (to stay under Step Functions' 25K event limit), and invokes the Launcher Lambda for each batch.

↓
2

Launcher Lambda

Python + Boto3

Scans the S3 bucket for video files, builds chunk definitions (each video divided into 60-second chunks starting at the 5-minute mark), and starts a Step Functions execution.

↓
3

Step Functions State Machine

AWS Step Functions (Nested Map)

Orchestrates parallel processing: outer Map iterates videos (20 concurrent), inner Map iterates chunks per video (40 concurrent workers). Includes retry logic with exponential backoff.

↓
4

Worker Lambda

Python + OpenCV (Docker, ARM64)

Reads a 60-second chunk of video via presigned URL. Performs coarse 1fps black frame scan, then refines detections to frame-accurate boundaries. Returns timestamped black frame runs.

↓
5

Merger Lambda

Python + Boto3

Collects all chunk results, merges adjacent runs, extracts discrete events, pairs them into commercial segments, probes for missing partners, and writes final results to S3 and DynamoDB.

Infrastructure Resources

S3 Bucket

Stores source videos (video/) and processing results (results/). Videos organized by network in subfolders.

DynamoDB Tables

Three tables: chunk results, per-video segment results (video_name + timestamp), and evaluation summaries.

Lambda Functions

Containerized (Docker) on ARM64. Worker: 2GB RAM, 5 min timeout. Merger: 512MB, 15 min timeout. Launcher: 256MB, 1 min.

Step Functions

Standard workflow with nested Map states. Processes up to 800 chunks concurrently (20 videos × 40 chunks).

Detection Algorithm

Step 1: Black Frame Detection (Worker)

Each 60-second chunk is analyzed for black frame runs using color histogram comparison:

1. Compute 32-bin color histogram for each frame (RGB channels)
2. Calculate chi-squared distance from a pure-black reference histogram
3. If distance < 0.3 → confirmed black frame
4. If distance < 8.0 → suspicious (may be very dark scene)
5. Refine coarse detections to frame-accurate start/end timestamps

For normal chunks, a coarse 1fps scan finds candidates, then a frame-by-frame refinement pass in ±2s windows determines precise boundaries. Short probe windows (≤15s) use frame-by-frame scanning directly.

Step 2: Event Extraction (Merger)

Black frame runs from all chunks are merged and converted into discrete "events":

1. Merge adjacent runs across chunk boundaries (gap < 0.1s)
2. Filter runs shorter than 0.1s (noise)
3. Extract events: runs separated by ≥15 seconds become distinct events
4. Remove events in first 5 minutes and last 60 seconds (margins)

Step 3: Commercial Segment Pairing (Merger)

Events are paired into commercial segments using duration heuristics:

1. For each unpaired event, find the best partner where:
   - Duration is between 90s and 300s (1.5 to 5 minutes)
   - Duration is closest to a standard multiple of 30s
2. Remove "interior" events (black frames within paired segments)
3. Re-pair after cleanup
4. Probe for missing partners at standard offsets (30s, 60s, ..., 300s)
5. Short segments (<90s) → attempt to extend from endpoints
6. Final cleanup: merge overlaps, remove contained segments

Key Design Decisions

  • 5-minute margin: First 300s of each broadcast is skipped (pre-show content, network bumpers)
  • Histogram-based detection: More robust than simple pixel averaging — handles noise, compression artifacts, and near-black scenes
  • Standard duration matching: Commercial breaks are reliably multiples of 30 seconds, providing a strong pairing heuristic
  • Active probing: When events can't be paired from the initial scan, targeted Lambda invocations search for missing partners at expected offsets
  • 60-second chunks: Balances Lambda execution time against parallelism and Step Functions event limits

Color and Region-Based Visual Differences Detection Example

An alternative visual detection method for networks like CNN and Fox News that don't have reliable black frames between commercials. Instead of looking for black frames, this approach detects the absence of the on-screen news ticker/chyron — when the ticker disappears, you're watching a commercial.

📊

Color Profile Matching

A precomputed HSV color histogram captures what the ticker region looks like during news. Each frame is compared against this profile using chi-squared distance.

✂️

Region Cropping

Only the ticker region of each frame is analyzed (configurable crop boundaries), making detection fast and focused on the most informative pixels.

⚡

Single-Pass Scan

Processes the entire video in one pass at configurable frame rate (0.5–1 fps), grouping consecutive dissimilar frames into commercial segments.

🔄

Automatic Fallback

If ticker detection finds zero segments, the system automatically falls back to the black-frame method.

How It Works

1. Crop each frame to the ticker region (e.g., bottom 25% of frame)
2. Convert to HSV and compute a 36×32×32 bin histogram
3. L1-normalize the histogram
4. Compare against the precomputed network color profile using chi-squared distance
5. If distance > threshold → frame is "dissimilar" (no ticker = commercial)
6. Group consecutive dissimilar frames into segments
7. Filter: remove short bursts (<10s), merge nearby segments (<60s gap)
8. Skip first 5 minutes and last 60 seconds (margins)

Per-Source Configuration

Each video source has its own detection profile stored in S3, configured via a JSON file:

{
  "CNN": {
    "crop_top_fraction": 0.75,     // Top of ticker region (75% down)
    "crop_bottom_fraction": 1.0,   // Bottom of ticker region (100%)
    "crop_left_fraction": 0.0,     // Full width
    "crop_right_fraction": 1.0,
    "chi_square_threshold": 0.35,  // Sensitivity
    "scan_fps": 0.5,               // Frames per second to sample
    "profile_key": "config/profiles/CNN_profile.npy"
  }
}

Threshold tuning: lower = fewer detections (less sensitive), higher = more detections (more false positives). The profile must match the era of the video — a 2025 ticker profile won't work on 2005 footage.

Building a Color Profile

Profiles are built from ground-truth-labeled videos. The profile builder:

1. Load ground truth CSV with labeled segments (news vs commercial)
2. For each "news" segment, extract frames from the ticker region
3. Compute HSV histograms for all news frames
4. Average all histograms into a single representative profile
5. Save as .npy file → upload to S3

The resulting profile represents "what the ticker looks like when news is airing." Frames that deviate significantly from this profile are classified as commercials.

Detection Routing

The video dispatcher automatically selects the detection method based on the video source:

Video uploaded → Dispatcher extracts source ID from filename
  → If source has a ticker profile configured → Ticker Detection
    → If 0 segments found → Fallback to Black-Frame Detection
  → Otherwise → Black-Frame Detection (default)

Transcription & AI Segmentation Pipeline

A separate automated pipeline uses the commercial detection output as input to an AI-powered transcript segmentation system. This identifies individual news segments (stories, interviews, analysis) within the non-commercial portions of each broadcast.

How It Works

1

Video Transcription

AWS Transcribe Lambda

Uploads trigger automatic transcription via AWS Transcribe, producing word-level timestamps for the entire broadcast. Output: JSON with every word and its precise start/end time.

↓
2

Commercial Filtering

Python (filter_commercials.py)

Uses the _segments.json output from the Commercial Segment Detector to remove all words that fall within detected commercial breaks. Inserts [ COMMERCIAL BREAK ] markers as hard boundaries.

↓
3

AI Segmentation

Amazon Bedrock (Claude Sonnet)

The filtered transcript is sent to Claude via Pydantic AI. The model acts as a "Junior Archivist," identifying news segment boundaries by detecting topic changes, anchor handoffs, and reporter transitions. Returns structured segment data with verbatim first/last sentences.

↓
4

Timestamp Matching

Python (timestamp_matcher.py)

Maps the AI-identified segment boundaries (first/last sentences) back to precise video timestamps using the original word-level transcription data. Produces frame-accurate start/end times for each news segment. Output: news_results/{video}_segments.json

↓
5

Teaser Detection

Amazon Bedrock (Claude) + Python

Independently loads commercial segments (from VIDEO_BUCKET) and news segments (from PROCESSING_BUCKET). Merges them into a unified N/C timeline and uses Claude to detect teaser speech at N→C transitions. Output: teaser_results/{video}_with_teasers.json

↓
6

Evaluation

Python (evaluation_metrics.py)

Compares predicted segments against human-annotated ground truth using frame accuracy, segment F1 at multiple IoU thresholds, boundary F1 at multiple tolerances, and normalized edit distance. Runs before the merger to ensure evaluation data is available.

↓
7

Results Merger

Python + Boto3

Uses the teaser JSON as the authoritative timeline. Enriches N segments with titles from AI segmentation and full transcripts from the transcription bucket. Falls back to combining separate sources only if teaser JSON doesn't exist. Output: result/{video}.json

AI Segmentation Approach

The system uses Claude (via Amazon Bedrock) with a carefully crafted archival prompt that instructs the model to:

  • Respect hard boundaries: [ COMMERCIAL BREAK ] markers are absolute — no segment spans across them
  • Identify natural transitions: Anchor handoffs, reporter sign-offs, topic changes, and shifts in focus
  • Extract verbatim sentences: First and last sentences of each segment are extracted word-for-word for precise timestamp matching
  • Maintain continuity: Panel discussions that shift topics stay as one segment; same-story coverage across speakers stays unified
  • Complete coverage: Every part of the transcript must be assigned to a segment with no gaps

Evaluation Metrics

Frame Accuracy

Per-second classification accuracy — what percentage of seconds are correctly assigned to the right segment type.

Segment F1

Measures segment detection at IoU thresholds (0.25, 0.5, 0.75). A predicted segment must overlap sufficiently with ground truth to count as a match.

Boundary F1

Measures boundary detection accuracy at multiple tolerances (2s, 5s, 10s). How close are predicted boundaries to actual segment transitions?

Edit Distance

Normalized edit distance between predicted and ground truth segment type sequences — captures ordering and structural errors.

Serverless Deployment

The pipeline deploys as event-driven Lambda functions connected by S3 triggers:

Upload video to S3
  → [S3 trigger] → TVNewsTranscriptionLambda (AWS Transcribe)
    → [Readiness Checker] → tv-news-segmentation (Bedrock Claude)
      → news_results/{video}_segments.json
        → [S3 trigger on news_results/] → Fanout → tv-news-teaser-detector
          (loads commercial_results/ from VIDEO_BUCKET + news_results/ from PROCESSING_BUCKET)
          → teaser_results/{video}_with_teasers.json
            → [S3 trigger on .json suffix] → Evaluation (first)
              → Results Merger (enriches with titles + transcripts)
                → result/{video}.json → Web UI

End-to-End Data Flow

📤

1. Video Upload

Hour-long TV news broadcasts (.mp4) are uploaded to S3 organized by network (MSNBC/, CNN/, FNC/)

🚀

2. Batch Launch

run_batches.sh lists videos, groups into batches of 20, invokes the Launcher Lambda for each batch

⚡

3. Parallel Processing

Step Functions fans out: up to 800 concurrent Worker invocations scanning for black frames in 60s chunks

🔗

4. Merge & Pair

Merger Lambda collects results, pairs black frame events into commercial segments, probes for missing boundaries

💾

5. Store Commercial Results

Commercial segments written to S3 (commercial_results/{video}_segments.json) and DynamoDB.

📝

6. Transcription & AI Segmentation

Videos are transcribed (AWS Transcribe), commercial speech is filtered out using the detected segments, and Claude identifies individual news story boundaries. Output: news_results/{video}_segments.json

🎯

7. Teaser Detection

Teaser Detector loads both commercial and news segments, merges into a unified N/C timeline, and detects teaser speech at N→C transitions. Output: teaser_results/{video}_with_teasers.json

📊

8. Evaluation & Merge

Evaluation runs first (compares to ground truth), then Results Merger produces unified JSON enriched with segment titles and full transcripts. Output: result/{video}.json

🌐

9. Visualization

Staff access the web app, select a video, view detected segments on an interactive timeline, and correct any errors

✅

10. Validation

Human-corrected annotations saved back to S3 for ground truth building and model evaluation
